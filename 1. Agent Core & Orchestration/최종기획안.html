프로젝트 개요 & 문제 정의

Overview & Problem Statement

1.1 한 줄 정의

사용자가 읽고 있는 텍스트를 AI가 분석·재구성하고, 읽기 행동 데이터를 실시간 분석해 맞춤 개입(강조·쉬운 설명·퀴즈·요약)을 수행하여 끝까지 읽고 이해하도록 돕는 개인 맞춤형 AI 리터러시 코치.

1.2 문제 정의 — 왜 지금 필요한가

문제근거 / 현황
문해력 저하OECD PIAAC 기준 한국 언어능력 점수 273점(2012) → 249점(2023)으로 24점 하락. 27개국 평균보다 13점 낮은 수준으로, 2012년 평균 대비 +4점이었던 것에서 역전됨.
숏폼 중독·집중력 저하짧은 콘텐츠 중심 소비로 긴 글을 끝까지 읽고 이해하는 능력 약화. 학습·업무 수행 능력 저하로 연결.
ADHD·난독증 사각지대집중 유지·해독·이해에 추가 지원 필요. 기존 도구는 난독증 지원/집중 관리/요약이 파편화되어 통합 서비스가 공백. 대부분 영어 기반이라 한국어 환자용은 전무.
기존 AI의 한계ChatGPT·Gemini·Claude는 텍스트를 대신 읽어 요약할 뿐, 사용자가 실제로 읽었는지·이해했는지·향상됐는지는 관리하지 않음.

1.3 우리의 해결 방향

AI가 내용을 대신 읽어주는 것이 아니라, 사용자가 스스로 읽고 이해하도록 돕는다. 읽기 행동·집중 상태를 실시간 분석하고 상황별 개입을 제공하여 독해 경험을 개선하고, 누적 데이터로 장기 문해력 성장에 기여한다.

근거 연구: AI 주석 기반 텍스트 제시(LARF, CHI 2025)에서 AI 주석 텍스트를 읽은 참가자는 세부 내용 기억에서 기존 대비 18.3% 높은 점수를 기록, 읽기 이해도·만족도 모두 유의미하게 향상. 본 시스템은 이를 한국어 환경에서 실시간 구현.

핵심 차별화 전략

Differentiation — "Isn't this just ChatGPT?"

⚠ 이번 기획의 최대 리스크

심사위원 질문 "ChatGPT에 PDF 넣고 요약시키면 끝 아닌가요?" — 이 한 방에 무너지면 안 된다. 답은 기능 나열이 아니라 구조(폐루프)로 한다.

2.1 핵심 프레임

GPT는 텍스트를 처리하고, 우리는 사람을 관리한다.

2.2 폐루프(Closed-Loop) 구조 — 차별화의 본질

① 측정 (Measure)
읽기 행동(스크롤·체류·이탈) + 이해도(퀴즈)로 인지부하·집중도 정량화
② 개입 (Intervene)
상태에 따라 강조·쉬운 설명·넛지·즉석 퀴즈로 재집중·이해 유도
③ 추적 (Track)
누적 시계열로 Literacy Score 성장 곡선 산출 + 프로필 갱신
↻   ③의 프로필이 다시 ①②의 개인화에 반영되는 순환 구조   ↻

2.3 ChatGPT vs 본 서비스

기준ChatGPT / 일반 AI본 서비스

향상됐는지 추적✕ 세션 휘발✓ 누적 시계열 → 성장 곡선
개인화△ 매번 초기화✓ Dynamic Literacy Profile 누적

목적정보를 대신 처리사용자가 스스로 성장
한 줄 방어: "ChatGPT는 한 번 읽고 잊지만, 우리 시스템은 사용자가 어떻게 읽는지 측정하고, 이해했는지 검증하고, 나아지는지 평생 추적합니다."

진입 전략: 무거운 앱 설치·업로드 대신 크롬 확장으로 마찰을 0에 가깝게 낮춰, "평소 읽던 뉴스·논문 위에서 바로 켜본다"는 낮은 진입장벽으로 첫 사용을 유도한다.

핵심 지표: Literacy Score 설계

The Heart of This Project

차별화를 눈에 보이게 만드는 유일한 장치. "문해력이 향상됐다"는 주장을 단일 정량 지표 + 전후 비교 그래프로 증명한다.

4.1 지표 구성

Literacy Score = f( 이해도, 집중도, 난이도 보정 )
  ① Comprehension (결과지표) = 퀴즈 정답률 ÷ 지문 난이도(한국어 가독성 지수)로 정규화
  ② Engagement (과정지표) = 스크롤 속도·체류시간·이탈률 → 능동읽기(Deep) vs 훑기(Skim) 점수
  ③ Cross-Validation 보정 = 정답이어도 스크롤이 비정상적으로 빠르면(찍기 의심) 감점
  ④ Longitudinal = 단일 세션이 아닌 시계열 추세(Trend)로 성장 판정

→ 점수 엔진은 LLM이 아닌 순수 함수(결정적·재현성)로 구현한다. 같은 입력이면 같은 점수 = 심사 시연·검증에 유리.

4.2 두 가지 측정 차원

차원유형측정 방식
실시간 집중력Process Metric인지부하(Cognitive Load): 스크롤 속도·문단 체류 시간·페이지 이탈률 → 능동/수동 읽기 실시간 점수화
실질적 이해력Outcome Metric인터랙티브 퀴즈: 난이도 × 정답률 → '텍스트 복잡도 대비 이해도(Comprehension Score)'

4.3 신뢰성 근거 (추가 질문 대비 키워드)

  • Flesch-Kincaid(한국어 대응) 가독성 지수: 원문 난이도를 객관 정의하고 점수를 보정(Normalization)해 공정 비교.
  • Cross-Validation: 퀴즈 정답 + 스크롤 패턴 교차 검증으로 '찍기' 배제, 데이터 신뢰도 향상.
  • Longitudinal Study: 수주간 누적 시계열로 추세 분석 → 일시적 오류 배제.

❌ 절대 하지 말아야 할 말 (과장 = 거짓)

  • "의사·전문가의 임상 시험을 거쳤습니다" — 실제로 안 했으므로 거짓.
  • "뇌파(EEG)를 측정합니다" — 하드웨어 없음, 불가능.
  • "사용자가 좋아하니까 향상된 겁니다" — 주관적 느낌은 과학적 근거 아님.
핵심 요약 답변: "우리는 읽는 과정(행동 데이터)과 읽은 결과(퀴즈 데이터)를 결합해 인지부하와 이해도를 정량화함으로써 문해력 향상을 과학적으로 추적합니다."

5.1 동적 호출 메커니즘 (StateGraph)

사용자 상태 변화에 따라 에이전트 간 전이(Transition) 발생:

Content Reducer → Cognitive Care → Reward Agent → Literacy Profile Agent 순의 유기적 호출, 오케스트레이터가 공유 상태(Shared State) 기반으로 라우팅.

5.2 핵심 설계 원칙 — 입력원 독립

오케스트레이터는 이벤트의 출처를 가리지 않는다. 입력은 reading_events(scroll/pause/blur/focus) + 글 텍스트, 출력은 개입 명령(넛지·하이라이트)과 세션 결과(score)뿐이다. 이 이벤트가 웹앱에서 오든 크롬 확장에서 오든 PDF 뷰어에서 오든 동일하다. → 확장 추가는 "코어 교체"가 아니라 "새 입력원 추가"이며, 기존 계약·변환 어댑터를 거의 그대로 재사용한다. (상세 §9)

9.1 제품 형태 — 파일 업로드 없는 크롬 확장(주력)

웹 서핑 중 접하는 모든 텍스트(뉴스·블로그·논문·기사·PDF)에 실시간 개입하는 브라우저 확장 프로그램. on/off 토글로 켜두면 백그라운드에서 상시 동작하여, 사용자가 "보던 대로" 읽는 동안 측정·개입·점수화한다.

컴포넌트역할설명
크롬 확장주력실제 읽기 측정 + 라이브 넛지 오버레이. 웹페이지 + PDF(pdf.js 뷰어) 모두 지원
웹 대시보드공용점수/성장/배지 확인 — 성장 그래프(데모 핵심 화면)
웹 읽기 화면폴백무설치 데모용 — 심사위원이 확장 미설치 시 시연

읽기 UX를 양쪽에서 중복 유지하지 않는다: 확장=주력 / 웹읽기=데모 폴백 / 대시보드=공용.

9.2 확장 아키텍처 (Manifest V3)

# extension/
├─ manifest.json          declarativeNetRequest + web_accessible_resources
├─ popup/                 on/off 토글 · 온보딩(동의) · 파일 열기
├─ background/            service worker: 세션 수명 · PDF 리다이렉트 규칙
├─ content/               웹페이지 주입(shared 어댑터)
├─ pdf/                   pdf.js 자체 뷰어(viewer.html·js·css)
├─ vendor/pdfjs/          pdf.mjs · pdf.worker.mjs (Apache-2.0 번들)
└─ shared/                # 웹·PDF 공용화(중복 제거)
   ├─ tracker.js          읽기행동 이벤트 캡처(정규화 스키마 방출)
   ├─ overlay.js          Shadow DOM 오버레이(넛지·배지·퀴즈·툴팁)
   └─ session_client.js   세션 수명 + REST 배치 flush 전송
MV3 현실: service worker는 상시 데몬이 아니라 이벤트로 자고 깬다. 탭/네비게이션 이벤트마다 깨어나 모니터링 → 체감은 "항상 켜짐". 상태는 chrome.storage에 두어 진짜 24/7 프로세스는 불필요.

9.3 확장 ↔ 백엔드 계약 (기존 계약 재사용)

엔드포인트역할
POST /api/session/start추출 본문 content[]로 세션 시작(웹=Readability / PDF=pdf.js). 응답 camelCase
POST /api/session/{id}/events읽기행동 이벤트 배치 flush → 응답에 실린 개입 렌더(REST 이벤트 구동)
GET /api/session/{id}/result전체 오케스트레이터 실행 → Literacy Score·배지·시계열 반환

→ 백엔드 관점에선 "웹이든 PDF든 content[] 받으면 끝" — 신규 계약 불필요. 전송방식은 REST로 확정(ADR-001, §9-8).

PDF 지원 — pdf.js 자체 뷰어

Same Measurement for PDF as Web

9.4 왜 기본 뷰어로는 안 되고, 왜 pdf.js면 되는가

크롬 기본 PDF뷰어는 PDFium(네이티브 플러그인)이 페이지를 "그림처럼" 그린다 → 확장이 글자·좌표·스크롤에 접근 불가 = 스크롤 속도·퀴즈·단어 뜻 전부 불가.

pdf.js(Mozilla)는 PDF를 JavaScript로 다시 그려 DOM으로 되돌린다. 페이지를 <canvas>로 그리고 그 위에 단어마다 <span> 텍스트 레이어를 좌표 맞춰 깐다 → PDF가 "일반 웹페이지"가 되어 웹과 똑같은 트래커/오버레이를 그대로 쓴다.

핵심: 사용자 경험은 기본 뷰어와 동일(PDF 링크 클릭 → 열림, 업로드 없음). 바뀌는 건 "누가 렌더하느냐"뿐 — 크롬 대신 우리가 렌더해 만질 수 있게 만든다.

읽는 방식스크롤 속도창 위 퀴즈단어 뜻사용자 경험
크롬 웹페이지✓✓✓그대로
크롬 기본 PDF뷰어✕✕✕그대로
우리 pdf.js 뷰어✓✓✓그대로(자동 전환)

9.5 동작 흐름

PDF 링크 클릭 (https://….pdf 또는 로컬 파일)
  └ declarativeNetRequest 규칙이 main_frame 요청을 가로챔
      → chrome-extension://<id>/pdf/viewer.html?file=<원본 URL>
          ├ pdf.js가 원본을 fetch → 페이지별 canvas + textLayer(<span>) 렌더
          ├ getTextContent()로 본문 추출 → content[] 로 POST /session/start
          ├ tracker(공용): 스크롤 속도·체류·blur/focus → 백엔드
          ├ overlay(공용): 넛지·퀴즈·단어뜻 툴팁
          └ 단어 hover → textLayer span에서 단어 추출 → 용어풀이 → 툴팁

로컬 PDF는 파일 피커 → ArrayBuffer로 브라우저 로컬 렌더(서버 저장 없음). 스캔(이미지) PDF는 글자 레이어가 없어 MVP 제외 → OCR(Tesseract.js·무료)은 후속.

9.6 세션 수명 (on/off + 백그라운드)

[확장 off] → 아무 동작 안 함
[확장 on]  → 페이지 로드 → content script 주입
   "읽을 만한 글" 판정? ─ 예 → N초 체류/스크롤 → 세션 시작 + 이벤트 스트리밍
       실시간 focus 하락 → 넛지 오버레이
       탭 이탈/닫기/visibility hidden 지속 → 세션 종료 → result → 대시보드 기록

원칙		새로 과금되는 요소 없음. pdf.js(Apache-2.0)·Tesseract.js(후속, Apache-2.0) 무료 번들, 유료 API/호스팅 금지, 데모는 stub로 무과금. LLM 호출이 유료라면 그건 기존 오케스트레이터 이슈이지 확장이 새로 만든 비용이 아님

범위 밖(후속 트랙): "크롬 밖 다른 뷰어·모든 앱 창" 측정은 네이티브 데스크톱 에이전트 = 별도 트랙.

심사 예상 질문 & 방어

Q&A Defense
Q1. ChatGPT에 PDF 넣고 요약시키면 끝 아닌가요?
GPT는 한 번 읽고 잊습니다. 우리는 어떻게 읽는지 측정하고, 이해했는지 검증하고, 나아지는지 추적하는 폐루프 시스템입니다. 게다가 붙여넣기 없이 크롬에서 보던 대로 측정합니다.
Q2. 문해력 향상을 정말 측정할 수 있나요? (임상적으로)
읽는 과정(행동 데이터)과 읽은 결과(퀴즈 데이터)를 결합해 인지부하·이해도를 정량화하고, 시계열 추세로 성장을 추적합니다. (EEG·임상시험 주장 금지)
Q3. 그 점수를 신뢰할 수 있나요?
Flesch-Kincaid 가독성 보정으로 공정 비교, Cross-Validation으로 '찍기' 배제, Longitudinal로 일시 오류 배제. 점수 엔진은 LLM이 아닌 결정적 순수 함수라 재현됩니다.
Q4. 크롬 기본 뷰어는 PDF를 못 만진다던데, PDF는 어떻게 측정하나요?
기본 뷰어(PDFium)는 그림처럼 그려 접근 불가라, PDF 링크를 우리 pdf.js 뷰어로 자동 전환해 DOM으로 되돌립니다. 그러면 웹페이지와 똑같이 스크롤·퀴즈·단어 뜻이 동작합니다. 사용자 경험은 동일합니다.
Q5. 개인정보·로그인은요? 상시 감시 아닌가요?
로그인 없이 익명 기기 UUID만 씁니다. ON일 때 읽을 만한 글에서만 동작하고, 화면 상시 감시·카메라·크롬 밖 앱은 보지 않습니다. 문서는 로컬 처리(서버 저장 없음)입니다.
Q6. 환각(hallucination)은 어떻게 막나요?
용어풀이는 신뢰 출처에 RAG 그라운딩하고, Ragas Faithfulness로 정량 검증합니다.
