RAG 엔진(Role 2) 팀원 전달용 피드백 및 개선 요청 사항
크롬 확장 프로그램을 실제 뉴스 기사에 연동해 테스트해본 결과, RAG 용어 사전 엔진 쪽에 다음과 같은 한계점들이 발견되었습니다. RAG 파트 팀원분께 이 내용을 전달해 주시면 더 완벽한 AI 케어 에이전트가 될 것입니다!

🚨 현재 RAG 엔진의 주요 문제점
1. 정적 JSON 데이터베이스에 과도하게 의존 (유연성 부족)
증상: 현재 data/term_dictionary.json에 미리 등록되어 있는 단어(주로 IT 용어)가 아니면 무조건 "뜻을 찾지 못했습니다"라고 반환합니다.
문제점: 사용자가 읽는 기사는 경제, 사회, 문화 등 무궁무진한데, 모든 단어를 JSON 파일에 미리 하드코딩해 둘 수는 없습니다.
2. 미등록 단어에 대한 동적 LLM 추론(Fallback) 부재
증상: 사전에 없는 단어 요청이 오면 바로 검색 실패 처리(source: not_found)를 해버립니다.
문제점: 명색이 AI 리터러시 에이전트인데, DB에 없는 단어라도 현재 읽고 있는 기사 본문(Context)을 바탕으로 LLM이 실시간으로 의미를 유추해서 설명해주는 기능이 없습니다.
3. 복합 명사 및 조사 처리(형태소 분석) 한계
증상: 사용자가 "주가 누르기 방지법을"처럼 띄어쓰기나 조사가 포함된 구절을 드래그하면, 이를 통째로 하나의 키워드로 인식해버려서 매칭에 실패합니다.
문제점: 형태소 분석기(예: Kiwi, KoNLPy)나 키워드 추출 로직을 통해 "주가 누르기", "방지법" 등 핵심 명사만 발라내어 검색하는 전처리 과정이 부족합니다.
🛠️ RAG 팀에 요청하는 3단계 개선 방안 (Action Items)
[Step 1] 동적 LLM 답변 생성 로직 추가 (필수)

JSON DB 검색(lookup_term) 결과가 없을 경우, 에러를 뱉지 말고 **LLM(GPT-4o 등)에게 "기사 문맥(Context)을 고려하여 '{word}'의 뜻을 50자 이내로 설명해줘"**라고 프롬프트를 보내 실시간으로 답변을 생성하도록 코드를 수정해 주세요.
[Step 2] 외부 외부 백과사전 API 연동 (권장)

JSON DB 외에도 '국립국어원 우리말샘 API'나 '위키백과 API'를 2차 검색 출처로 연동하여 환각(Hallucination) 없는 공신력 있는 데이터를 먼저 가져오도록 개선해 주세요.
[Step 3] 검색어 전처리(형태소 분석) 도입 (고도화)

사용자가 드래그한 텍스트에서 특수문자, 불필요한 조사("~은", "~를")를 제거하고 핵심 명사 위주로 검색할 수 있도록 자연어 전처리 파이프라인을 추가해 주세요.
