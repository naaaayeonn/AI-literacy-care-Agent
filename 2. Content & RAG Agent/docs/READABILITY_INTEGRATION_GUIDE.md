# [팀원 전파용] 한국어 이독성(Readability) 및 난이도(Difficulty) 독립변수 개편 가이드

이 문서는 콘텐츠 에이전트(2번)가 적용한 **이독성(표면적 가독성)** 및 **난이도(개념적 복합성)** 산출 공식 개편 사항을 타 팀원(1번 Orchestrator, 3번 Backend, 4번 Frontend)에게 공유하고 조율하기 위해 작성되었습니다.

---

## 1. 개편 핵심 요약
기존에는 `난이도 = 100 - 이독성`이라는 단순한 대칭 관계(종속변수)를 사용했으나, 교수님 피드백 및 설계 고도화를 통해 **이독성과 난이도를 각각의 고유한 피처를 사용하는 독립변수**로 분리하였습니다.

* **이독성 (Readability)**: 글이 형태적·문법적으로 잘 읽히는가? (표면적 요인)
* **난이도 (Difficulty)**: 글이 다루는 개념과 주제가 인지적으로 얼마나 무거운가? (개념적 요인)

---

## 2. 최종 적용 공식

### 2.1 피처 정의
1. **평균 어절 길이** = `전체 글자 수(공백 제외) / 전체 어절 수`
2. **평균 어절 수** = `전체 어절 수 / 전체 문장 수`
3. **고급 어휘 비율(%)** = `(3등급 이상 + 미등록 어휘 개수) / 전체 한글 어휘 수 * 100` (국립국어원 2024 기초 어휘 목록 대조)
4. **전문 용어 밀도(%)** = `(표준 전문 용어 매칭 개수 / 전체 어절 수) * 100` (국어기본법 고시 표준 전문 용어 대조)

> [!NOTE]
> 형태소 분석기 없이 고속 연산을 처리하기 위해, **한국어 조사/어미 분리 알고리즘**을 적용하여 어간을 정밀 분리한 뒤 사전에 매칭하는 결정론적 알고리즘을 사용합니다. (AI 환각 0%, 지연 시간 0ms 수준)

### 2.2 이독성 (Readability Score) 공식
$$\text{Readability Score} = 95.0 - (\text{고급어휘비율} \times 0.35) - (\text{평균어절수} \times 1.0) - (\text{평균어절길이} \times 2.0)$$
* **스케일링**: 원점수 유효 범위인 `15.0 ~ 85.0`을 `0.0 ~ 100.0` 전 구간으로 선형 보정(Min-Max Scaling)하여 분산 폭을 극대화함.
* **해석**: 점수가 100점에 가까울수록 문장이 짧고 쉬운 어휘로 구성되어 읽기 편합니다.

### 2.3 난이도 (Difficulty Score) 공식
$$\text{Difficulty Score} = 100 \times \left(1 - e^{-0.25 \times \text{전문용어밀도(\%)}}\right)$$
* **스케일링**: 표준 전문 용어가 1개라도 출현하면 난이도가 크게 점핑하며, 인지적 부하(Fechner의 법칙)에 따라 비선형적으로 난이도가 100점까지 수렴해 가는 지수 스케일링 모델(Exponential Scaling)을 적용함.
* **해석**: 점수가 100점에 가까울수록 다루고 있는 주제에 법적/학술적 표준 전문 용어가 빽빽하게 담겨 있어 고도의 인지 부하를 유발합니다.

## 3. 예시 작동 시나리오

| 텍스트 예시 | 이독성 점수 | 난이도 점수 | 상태 진단 |
|:---|:---:|:---:|:---|
| **"양자역학은 입자의 상태를 확률적으로 설명한다."** | **96.2점** (높음) | **73.4점** (높음) 🚀 | 문장은 매우 매끄러우나, 개념적 전문 지식(양자역학)의 무게감을 지수 모델로 실감 나게 반영함. |
| **"나는 너고 너는 나고 그래서 그것은 내가 아닌 너인 동시에..."** | **42.0점** (낮음) | **0.0점** (없음) | 개념은 없으나, 문장 구조가 꼬이고 비문이어서 이독성이 깨짐. |

## 4. 팀원별 액션 아이템 (Action Items)

### 1번 Orchestrator (오케스트레이션 / 점수 엔진 담당)
* **API 호환성**: 2번 에이전트는 기존 계약 규격을 준수하기 위해 `calculate_difficulty_score(float_value)` 함수에 **이중 가드(Polymorphic Guard)**를 탑재했습니다. 
  * 만약 float 점수를 넘기면 하위 호환을 위해 `100 - 점수`를 뱉어내며 절대 에러를 유발하지 않습니다.
  * 다만, 독립변수 난이도를 사용하기 위해 `agent.py` 내부에서는 이제 텍스트 원문을 넘겨 난이도를 독립 산출하도록 2번 코드 레벨에서 업데이트를 완료했습니다.
* **테스트 코드 보정**: `test_content_e2e.py`에서 두 점수의 대칭 합산 100 제약 검증(`abs(readability + difficulty - 100) < 0.1`)은 제거하였으니 참고바랍니다.

### 3번 Backend (행동 데이터 / DB 담당)
* **세션 상태 병합**: `difficulty_score`와 `readability_score`가 더 이상 대칭 관계가 아니므로, DB 스키마나 로그에 두 값을 각각의 독립 칼럼으로 유의미하게 분리 적재하고 수집하여 주시기 바랍니다.

### 4번 Frontend (읽기 UI / 대시보드 담당)
* **시각화 다각화**: 대시보드 및 읽기 화면에서 두 점수가 독립적으로 제공됩니다.
  * **이독성(Readability)**: "문장의 매끄러움 / 읽기 피로도" 지표로 시각화 (예: 게이지 바).
  * **난이도(Difficulty)**: "개념의 복잡도 / 전문성 수준" 지표로 시각화 (예: 뇌 아이콘, 인지 난이도 단계).
  * 독자에게 "이 글은 문장은 쉬운데 전문 용어가 많아요" 또는 "주제는 쉬운데 글이 꼬여 있어요"와 같은 입체적인 피드백을 전달할 수 있습니다.
