# 기술 심사 및 발표 방어 스크립트 (DEFENSE_SCRIPT)

본 문서는 **2번 역할 (Content & RAG Agent)**이 구현한 핵심 기술적 의사결정 사항과 발표 및 심사위원 예상 질문에 대한 방어 논리를 정리합니다.

---

## 1. RAG 용어풀이 환각(Hallucination) 방지

### Q. 용어풀이 생성 시 LLM의 환각 현상을 어떻게 원천적으로 방지하나요?
- **답변:** 
  본 시스템의 RAG 엔진은 LLM을 사용해 용어의 정의를 **생성(Generation)하지 않고**, 신뢰할 수 있는 사전 데이터(로컬 단어장 및 국립국어원 우리말샘 오픈 API)에서 정의를 **조회(Retrieval)하여 그대로 인용**합니다.
  생성 과정을 배제하고 엄격한 매칭 및 인용 프로세스를 거치기 때문에 환각 현상이 발생할 확률은 **0%**이며, 신뢰성 검증 지표인 `faithfulness_score`는 항상 **1.0 (완전 무결)**을 유지합니다.

### Q. 로컬 단어장에 없는 새로운 전문 용어가 검색될 때는 어떻게 대응하나요?
- **답변:** 
  다층(Multi-tier) 탐색 구조를 사용합니다:
  1. **1단계 (로컬 사전)**: 수동으로 검증 및 수집된 106개의 IT/일반 핵심 단어장에서 대소문자 구분 없이 완벽 매칭을 시도합니다.
  2. **2단계 (공공 API)**: 국립국어원 **우리말샘 오픈 API**를 통해 최신 어휘 정의를 실시간으로 가져옵니다.
  3. **3단계 (RAG 유사도)**: 단어가 약간 변형되었을 경우를 대비해 `sentence-transformers`를 활용해 코사인 유사도가 0.3 이상인 로컬 용어를 매칭하여 유연하게 대처합니다.
  이 모든 탐색 과정이 실패할 경우에만 `source="not_found"`로 안전하게 가드하여 사용자 화면에 지저분한 정의가 출력되는 것을 방지합니다.

## 2. Gemini 2.0 Flash 전환 및 비용 0원 원칙

### Q. 당초 Claude 모델을 사용하다가 Gemini로 전환한 이유는 무엇인가요?
- **답변:** 
  경진대회 및 상용 서비스의 **비용 0원 원칙(Cost-Zero Principle)**을 고수하기 위해서입니다.
  쉬운 문장 재구성 및 퀴즈 생성 단계는 빈번한 LLM 호출이 필요한데, paid API(Claude) 대신 Google AI Studio에서 제공하는 무료 티어의 **`gemini-2.0-flash`** 모델로 완전 전환하여 API 사용료를 0원으로 통제했습니다.
  또한, Gemini 2.0 Flash는 매우 빠른 레이턴시(속도)를 제공하여 실시간 가독성 개선 인터랙션에 최적의 성능을 냅니다.

### Q. API 호출 제한(Rate Limit)이나 네트워크 장애가 발생하면 서비스가 마비되지 않나요?
- **답변:** 
  **Fallback Hardening(예외 복원력 강화)** 설계를 적용했습니다.
  네트워크 단절, API 키 누락, 또는 속도 제한이 감지되면 시스템은 즉시 **데모 폴백 모드** 또는 **원문 Passthrough 모드**로 자동 전환됩니다. 텍스트 재구성 실패 시 원문을 그대로 노출하고, 퀴즈 생성 실패 시 준비된 기본 퀴즈 데이터셋을 즉시 로드하여 서비스 흐름이 절대 중단되지 않도록 방어합니다.

## 3. 웹 및 PDF 인입 정규화와 노이즈 필터링

### Q. 크롬 확장 프로그램과 PDF 뷰어에서 텍스트를 가져올 때 생기는 레이아웃 노이즈는 어떻게 제거하나요?
- **답변:** 
  PDF 파싱 시 나타나는 논문 제목, 페이지 번호, 머리말/꼬리말(Header/Footer) 등의 반복적 노이즈를 제어하기 위해 **이중 임계값 필터링 알고리즘**을 도입했습니다:
  - **빈도 조건**: 전체 문단 중 동일한 텍스트가 3회 이상(또는 전체의 10% 이상) 중복 등장하는 경우
  - **길이 조건**: 50자 이하의 짧은 라인인 경우
  이 두 조건을 동시에 충족하는 텍스트 라인은 본문의 맥락을 깨트리는 페이지 노이즈로 간주하여 자동 제거 후 깨끗한 문단으로 재결합합니다. 이 과정을 통해 Chunker 및 LLM 입력의 토큰 낭비를 막고 가독성 분석의 왜곡을 방지합니다.

## 4. 가독성 점수(Readability Score) 산출 공식의 신뢰성

### Q. 가독성 및 난이도 점수는 어떤 수학적 근거로 계산되나요?
- **답변:** 
  Flesch-Kincaid 가독성 공식을 한국어의 언어적 특성에 맞춰 보정한 자체 **한국어 가독성 공식**을 사용합니다.
  - **문장 평균 어절 수 (평균 어절 길이 패널티)**
  - **어절 평균 음절 수 (단어의 복잡성 패널티)**
  - **전문 용어 비율 (영어 단어 및 학술 접미사 `~성`, `~율`, `~화` 등의 분포 비율 패널티)**
  위 3가지 차원을 가중 결합하여 0~100 사이의 수치로 산출하며, 난이도 점수는 `100 - 가독성 점수`로 역산해 직관적인 난이도(높을수록 어려움)를 반환합니다. 모든 결과값은 `0`과 `100`으로 클램핑되어 비정상적인 값(NaN 등)이 발생하지 않는 순수 함수 구조로 안정성을 기했습니다.
